# 妖怪コレクション ゲームデザイン

**運営**: 合同会社妖怪屋（代表: 相田一成）
**ミッション**: 地域の未活用資産（夜・宴会場・神域）を「妖怪×エンタメ×IT」で再定義し、新たな観光価値と収益を創出する

---

## 0. 本ドキュメントの位置づけ

- **現行コードベース（dev ブランチ）** = MVP として稼働済みの位置情報ゲーム
- **本ドキュメント** = 既存実装の整理 ＋ 機能拡張ロードマップ（陣営／寺社ポータル／賽銭／レベニュー分配）

設計の優先方針:

1. 既存メカニクスを破壊しない（後方互換）
2. スタンプラリー（`rally_key`）と寺社ポータル（`portal_id`）は **別機能として並走**させる（統合しない）
3. 既存「封印」体験を**祓い手陣営の挙動として再定義**し、超自然派陣営の体験を追加する形で双陣営化する

---

## 1. 現状の実装（Phase 0 / MVP）

### 1-1. ゲームループ

1. ユーザは地図を開く（Leaflet + OSM タイル、夜モードは CARTO dark）
2. ブラウザの Geolocation で現在地取得（取得失敗時は dev ビルドのみデバッグモード起動）
3. 半径 **20m** の捕獲圏に妖怪マーカー（hitodama 画像）が入ると封印解除モーダルが自動起動
4. 「図鑑に封じる」で `POST /capture` → サーバ側でも距離検証 → 図鑑に追加
5. 図鑑からは詳細閲覧と「妖怪カメラ」（リアカメラ映像に妖怪 PNG を AR 合成して撮影）

### 1-2. 主要エンティティ（DynamoDB）

| テーブル | キー | 主属性 |
|---|---|---|
| `kodama_db` (妖怪マスタ) | `yokai_id` (HASH) | name, kana, lat/lon, notes, appearance, images[], image_types[], regions[], category_tags[], keywords[], `rally_key`, `night_only` |
| `youkai-captures` (捕獲ログ) | `deviceId` (HASH) + `youkaiId` (RANGE) | capturedAt, userLat, userLon, rally_key |

### 1-3. 既存 API

| メソッド | パス | 用途 |
|---|---|---|
| GET | `/youkai` | 通常妖怪一覧（rally_key を持たないもの） |
| GET | `/youkai/{id}` | 妖怪詳細 |
| POST | `/capture` | 捕獲（封印）。サーバで 20m 検証 |
| GET | `/collection?deviceId=` | 自分の捕獲済み一覧 |
| GET | `/rally?key=` | ラリー妖怪一覧 |
| GET | `/rally/collection?deviceId=&key=` | ラリー進捗 |
| GET/POST/DELETE | `/admin/youkai*` | CMS（妖怪 CRUD + 画像 S3 アップロード） |

### 1-4. 既存仕様で押さえておくポイント

- **ユーザ識別は device-id Cookie のみ**（10年保持）。アカウント登録なし
- **スタンプラリー** は `rally_key` ベースの期間限定イベント機能として完成済み。Phase 2 以降も**この機構には手を入れず維持**
- **夜モード** は `IS_NIGHT` フラグ＋`night_only` 妖怪で実装済み（祟り系の伏線として活用可能）
- **AR 妖怪カメラ** は陣営化しても陣営共通の体験として維持できる
- インフラは AWS SAM（API Gateway + Lambda + DynamoDB + S3 + CloudFront）、フロントは静的 SPA

## 3. Phase 1: 二陣営対立モデル

### 3-1. 陣営

**祓い手陣営（秩序側）**
- 構成ロール: 陰陽師、祈祷師、巫女
- 目的: 妖怪を**封じる**
- 思想: 神道・仏教的世界観、調伏と鎮魂
- **既存「封印 → 図鑑に封じる」体験はそのまま祓い手のメインアクション**

**超自然派陣営（混沌側）**
- 構成ロール: 妖術師、山伏、呪術師
- 目的: 妖怪を**見つけて仲間にする / 解放する**
- 思想: 民間信仰、修験道、付喪神文化
- 新規アクション: `release`（解放）／ `bond`（使役/契約）

※「払う」はお金を払うではなく **邪気を祓う** の意。

### 3-2. 既存実装との対応

| 既存 | Phase 1 後 |
|---|---|
| `POST /capture` | 祓い手の `seal` アクションに改称（API は後方互換維持） |
| `CaptureDBItem` | `actionType: 'seal' \| 'release' \| 'bond'` を追加。既存レコードは `seal` 扱い |
| 図鑑（collection） | 祓い手は「封印図鑑」、超自然派は「使役妖怪帳」（同じ DB を view 切替） |
| hitodama マーカー | 双陣営共通（同じ妖怪が双方の標的） |
| 妖怪カメラ | 共通体験。陣営別の AR フィルタ追加余地あり（札／呪符など） |
| device-id 識別 | `faction` を Cookie or 新ユーザ DB に保持。**陣営選択は opt-in**（未選択ユーザは祓い手挙動） |

### 3-3. PvP的緊張感

- 同じ妖怪に対し祓い手は封印、超自然派は使役を狙う → 同じ場所で目的が逆になる
- ただし MVP 段階では**他プレイヤーのアクションが自分の体験を阻害しない**設計（敵対 PvP ではなく目的対立 PvP）
- 将来: 寺社ポータル単位で陣営勢力マップを表示（陣営の優勢色で寺社マーカー着色）

### 3-4. 陣営別の詳細設計（別チャット統合 TODO）

以下は別チャットで設計済みの可能性あり。情報を統合する:

- [ ] 各陣営の詳細スキル/職能設計（陰陽師・祈祷師・巫女 vs 妖術師・山伏・呪術師の差別化）
- [ ] 妖怪の属性・分類体系（既存の `category_tags`/`keywords` 上に乗せる）
- [ ] 封印/解放/使役のゲームメカニクス詳細（成功率、コスト、クールタイム）
- [ ] 陣営別の世界観・ストーリーライン
- [ ] レベル/育成システム
- [ ] PvP/陣営戦の具体ルール

## 4. Phase 2: 寺社ポータル & 賽銭

### 4-1. ポータル共通設計

- ポータル（神社仏閣）は両陣営で共通
- 同じ場所で目的が逆になる → PvP 的緊張感
- 寺社側は陣営を問わず賽銭が落ちる構造（収益最大化）
- プレイヤーの陣営変更コストが低く、継続率向上

### 4-2. スタンプラリーとの関係: 別機能として並走

**重要: 既存の `rally_key` ベースのスタンプラリーには手を入れない。**
ポータルは独立した新機能として並走させる。

| 機能 | 役割 | キー | 性質 |
|---|---|---|---|
| **スタンプラリー（既存）** | 期間限定イベント単位の妖怪収集 | `rally_key`（イベントID） | 一過性／キー入力で参加／達成で受付提示 |
| **寺社ポータル（新規）** | 寺社単位の常設拠点。賽銭と陣営バフの場 | `portal_id` | 常設／位置ベースで自動認識／決済を伴う |

設計上の意図:
- スタンプラリーは**自治体・観光イベント向けの軽量集客機構**として完成しており、そのまま価値がある
- 寺社ポータルは**法的に重い（決済・宗教法人契約）常設インフラ**で、設計トレードオフが全く違う
- 両者を同居させる: 旗艦寺社が「常設ポータル」かつ「期間限定ラリー会場」になることはあり得るが、内部的には別エンティティ／別 API として扱う
- `kodama_db.rally_key` 属性はそのまま維持。妖怪は portal にも rally にも独立に紐づけられる（多重所属可）

### 4-3. 新規エンティティ

```
PortalDBItem
  portal_id: string          # 主キー
  name: string                # 寺社名
  denomination: string        # 宗派/系統
  lat, lon: number
  owner_org_id: string        # 宗教法人ID（契約管理用）
  yokai_ids: string[]         # 紐づく妖怪
  buff_config: {...}          # 賽銭バフ設定（陣営別係数）
  active: boolean

DonationDBItem
  deviceId + createdAt        # 複合キー
  portal_id, amount, faction
  stripe_payment_intent_id
  share_to_temple / platform_fee / stripe_fee  # 内訳明示

PortalBuffDBItem
  deviceId + portal_id        # 複合キー
  expires_at, factor          # 賽銭バフの時限効果
  buff_kind: 'seal' | 'release' | 'encounter' | ...
```

### 4-4. 賽銭システム

**金額設定**
- **賽銭上限: 1回 1,000円**
  - 賭博性・景品表示法リスクの回避
  - 健全性の担保（家族・未成年でも安心）
  - リアル賽銭の感覚と一致

**推奨刻み**: 5円 / 50円 / 100円 / 500円 / 1,000円（45円 = 始終ご縁 等の語呂合わせも検討）

**効果カーブ**: log カーブで設計。5円でも効果はあるが、1,000円が最も効率的（200倍差はつけない）。

**陣営別効果分岐**

| 賽銭する陣営 | 効果 |
|---|---|
| 祓い手陣営 | そのポータルでの**封印コスト減/成功率＋** |
| 超自然派陣営 | そのポータルでの**解放/使役コスト減** |

宗教的整合性: 両者とも「**鎮める**ための奉納」と説明統一。神道の荒魂/和魂の発想と相性が良い。

**制限設計（要詰め）**
- [ ] 1ポータルあたりの1日賽銭回数制限
- [ ] クールタイム設定
- [ ] 月間上限 or 警告システム
- [ ] マルチアカウント対策（1端末1陣営縛り、陣営変更クールタイム等）

### 4-5. ユーザ識別の堅牢化

device-id 単独識別では賽銭/陣営の整合性が保てない。Phase 2 で導入:
- メアド or SNS連携（Apple/Google sign-in 推奨）
- 既存 device-id ユーザはサインイン時にマージ

### 4-6. ユーザ恩恵設計（賽銭インセンティブ）

1. **ご利益バフ** — 妖怪遭遇率アップ、レア妖怪出現、捕獲/封印/解放成功率＋（神社ごとに固有効果）
2. **デジタル御守の発行** — 賽銭額に応じたインベントリアイテム＋バフ効果
3. **限定妖怪の解放** — 寺社の伝承に紐づく固有妖怪（伏見稲荷→九尾、出雲→八岐大蛇等）
4. **おみくじガチャ** — 賽銭500円でアプリ内おみくじ、結果でレア確定チケット
5. **奉納ランキング** — 月間ランキング、上位者に称号・限定スキン（氏子文化の現代版）
6. **絵馬機能** — 賽銭時に願い事＋スタンプ、他ユーザと共感ボタンで応援
7. **祓い・厄落とし** — 「祟り属性」妖怪を賽銭で浄化（祓い手陣営向け。既存 `night_only` 属性を活用可能）
8. **遠隔参拝** — 物理移動できない人の推し神社賽銭（コロナ後需要、海外在住日本人需要）

設計思想: ゲーム課金は「自分への投資」、賽銭は「寺社への貢献」という体験設計。同じ金額でも罪悪感が減り満足度が上がる。バフは時間経過で切れる → 自然なリピート発生。

## 5. Phase 3: 収益分配モデル & 寺社関係構築

### 5-1. 法的建付け

**OK: ユーザが寺社に寄付する。妖怪屋は決済・プラットフォーム提供の対価として手数料を取る。**

NG: ユーザが妖怪屋に払う → その一部を寺社に分配する（宗教法人非課税枠が使えない＋優良誤認リスク）

技術的には **Stripe Connect**（Standard or Express）で寺社を connected account 化し、決済時に application fee として手数料を引く構造が最も整合的。

### 5-2. 配分案

| 配分先 | 比率 |
|---|---|
| 寺社 | 70〜85% |
| 妖怪屋（プラットフォーム手数料） | 10〜20% |
| 決済代行（Stripe等） | 3〜4% |

参考: GoFundMe 2.9%、CAMPFIRE 17%、ふるさとチョイス（自治体課金型）。**15%前後**が業界相場として説明しやすい。

### 5-3. ユーザ透明性（必須）

賽銭画面で内訳を明示:

```
賽銭 1,000円
├ ◯◯神社へ  850円
├ 決済手数料   30円
└ プラットフォーム利用料  120円
```

実装上は `DonationDBItem` に share/fee 内訳を保存し、レシート画面でそのまま表示。

### 5-4. 寺社への説明話法

「賽銭の◯%を抜く」ではなく「**集客・アプリ開発・運用保守を一括で巻き取る対価**として◯%」と説明。IT 導入の業務委託料として理解しやすい構造。

### 5-5. 段階展開戦略

**フェーズ A: 旗艦寺社 10〜30 社と個別深掘り契約**
- 妖怪伝承が強い場所を選定（出雲、大江山、遠野、境港周辺、京都百鬼夜行系等）
- 書面契約、レベシェア・夜間拝観・限定御朱印まで作り込む
- **会津での1社目**: 相田代表の地元コネクションを活かす旗艦事例化

**フェーズ B: 標準契約書テンプレ＋簡易許諾**
- 旗艦事例ベースの標準契約で中規模寺社 100〜500 社展開

**フェーズ C: 申請プラットフォーム化**
- 寺社側から「うちもポータルにしてほしい」と申請できる Web フォーム
- 簡易審査＋電子契約

### 5-6. 大原則: 個別許諾必須

- 包括契約・神社本庁経由の一括は構造的に不可能
- 宗教法人ごとに事情が違う
- ポケモン GO 訴訟前例あり、無許可ポータル化は訴訟リスク
- 賽銭が絡むと寺社側の責任が重くなる
- 代替わりリスクに備え「個人」ではなく「法人」と契約

### 5-7. アプローチルート

- **神社系**: 神社本庁の地方支部、各県神社庁に「相談」スタンスで
- **仏教系**: 宗派の宗務庁単位（観光寺院多い宗派：臨済宗、天台宗、真言宗）
- **自治体経由**: 市町村観光課を巻き込み、地方創生×IT の看板で動く
- **既存妖怪縁の自治体**: 境港（鳥取）、遠野（岩手）、福崎（兵庫）、大江山（京都）

### 5-8. 寺社収益化メニュー（許諾止まりにしない）

1. レベニューシェア（文化財保護協力金/寄進名目で非課税枠活用）
2. 夜間拝観チケット抱き合わせ
3. 限定御朱印・授与品コラボ
4. 法要・祈祷送客モデル（祈祷料 5,000〜10,000円）
5. 縁日・宴会場としての境内貸し（妖怪屋の本業活用）
6. ストーリー監修料・伝承提供料（神職を妖怪学アドバイザー委嘱）
7. 寄進プラットフォーム化（電子賽銭）
8. 共同事業体（LLP）化（重点寺社）

**推奨組み合わせ: 3＋4＋5**（寺社の既存ビジネスに乗せる形が宗教法人会計上クリーン）

## 6. リスク管理

### 6-1. 法務・権利

- 個別書面許諾＋宗教法人との契約（最優先）
- 既存 IP（水木プロ、自治体登録商標）との非衝突確認
- 改正個人情報保護法・GPS 追跡の同意取得（**現状の Geolocation 利用時にも要再点検**）
- 「賽銭ガチャ」設計は景表法・賭博罪グレー → 弁護士相談マスト

### 6-2. 寺社サイドの摩擦

- 参拝者と遊戯目的ユーザの動線・マナー衝突
- 御朱印文化との誤認防止
- 檀家・氏子総会の合意（宮司/住職だけの口約束は危険）

### 6-3. 安全・地域

- 夜間運営の保険（PL/イベント保険）
- 私有地・参道通行権、夜間立入の境内使用許可
- 地元警察・近隣住民への事前通知

### 6-4. レピュテーション

- 「神聖な場所の商業利用」批判
- 戦没者慰霊系・事故/死亡があった寺社は地雷
- インバウンド客のマナー違反時の運営責任

### 6-5. 事業継続

- 宮司/住職代替わり対応 → 法人契約・3〜5年契約

## 7. 互換性ポリシー（実装方針）

既存実装を壊さないための原則:

1. **DynamoDB スキーマは追加のみ**（既存属性は維持、新属性は optional）
2. **既存 API は後方互換**（`/capture` のセマンティクスは祓い手の `seal` として継続）
3. **陣営未選択ユーザ = 祓い手挙動**（既存ユーザ体験そのまま）
4. **スタンプラリー（`rally_key`）と寺社ポータル（`portal_id`）は別機能として並走**（統合・移行はしない）
5. **既存 device-id ユーザはサインイン時にマージ可能**（Phase 2 で導入）
